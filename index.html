📌 事件

事件监听器

-参数1表示事件，如click、mouseover、keydown
-参数2表示事件发生时调用的函数，多为匿名函数
按钮事件有foucus、blur、dblclick、mouseover、mouseout
移除事件监听器-removeEventListener()
另外两种注册事件处理程序的方法：
事件处理属性-如onclick
内联事件处理程序（不推荐）
事件对象：
有时在事件处理函数中，会看到一个参数，名称为 event、evt 或 e。被称为事件对象

🧨 事件冒泡

事件是从被点击的最内层元素冒出来的

stopPropagation()-当它在事件处理程序中被调用时，可以防止事件冒泡到其他元素
事件捕获：与事件冒泡顺序相反
事件捕获默认被禁用，要启用它，你必须在 addEventListener（） 里传递caption选项
如container.addEventListener(“click”, handleClick, { capture: true });
事件冒泡实用方法：
当用户与大量子元素中的任何一个进行交互时执行某些代码时，我们在它们的父元素上设置事件监听器，并让事件从子元素冒泡到父元素，而不是必须单独在每个子元素上设置事件监听器
target和currentTarget
target 指向事件最初被触发的元素，而 currentTarget 指向事件处理器被附加的元素

⚡ 同步编程

同步函数：当同步函数运行时不能做其他任何操作
    原因：JavaScript 程序是单线程的它一次只能做一件事：所以如果它在等待我们长时间运行的同步调用返回，它就不能做其他事情

🚀 异步编程

事件处理器（add Event Listener）也是异步函数
回调：一个传递给另一个函数的函数（不易处理嵌套，故现代异步API不常用）
承诺——异步编程基础
    链式承诺：
    const fetchPromise = fetch(
    "https://mdn.github.io/learning-area/javascript/apis/fetching-data/can-store/products.json",
    );
    fetchPromise.then((response) => {
        const jsonPromise = response.json();
        jsonPromise.then((data) => {
        console.log(data[0].name);
        });
    });
    重写：
    const fetchPromise = fetch(
    "https://mdn.github.io/learning-area/javascript/apis/fetching-data/can-store/products.json",
    );
    fetchPromise
        .then((response) => response.json())
        .then((data) => {
        console.log(data[0].name);
    });
    我们不必在第一个then()的处理程序中调用第二个then()，而是可以返回json()返回的promise，并在该返回值上调用第二个then()

    catch()方法——用于错误处理
        异步操作成功时调用传递至 then() 的处理程序，异步操作失败时调用catch()处理程序
    承诺三种状态：
        pending初始状态，操作尚未完成
        fulfilled操作成功，此时调用了 promise 的.then() 处理器
        rejected操作失败,此时调用承诺的 .catch() 处理程序
    Promise.all()方法，它接受一系列承诺，并返回一个承诺
    Promise.any()方法，实现一系列承诺中的任意一个，不在乎哪一个
    
    async：
        在函数开头添加async，使其成为异步函数
    await：
        使得代码在该点等待，直到Promise被解决，此时Promise的fulfilled值被视为返回值，或者rejected值被抛出

🌳 DOM 脚本

Web 浏览器重要部分

  • 窗口 —— Window
  • 导航器 —— Navigator
  • 文档 —— Document

DOM 树结构

  • 树中的每个条目称为一个节点（元素节点 or 文本节点）。
  • 关系术语：根节点、子节点、后裔节点、父节点、兄弟节点。

选择元素

要操作 DOM 中的元素，首先需要选中它并将该元素的引用存储在变量中。

如：const link = document.querySelector("a");

  • Document.querySelector()
  • Document.querySelectorAll()（现在最常用）
  • Document.getElementById()
  • Document.getElementsByTagName()

创建与添加节点

  • Document.createElement() —— 新建元素。
  • Node.appendChild() —— 先将引用存储在变量中，再传入变量使用，移到节点底部。
  • Document.createTextNode() —— 创建文本节点。

移除节点

  • Node.removeChild() —— 移除节点及其父节点的引用。
  • Element.remove() —— 基于自身引用删除节点。

操作 CSS 样式

  1. 获取样式表列表（过时方法）

    • 用 Document.styleSheets 获取所有附加到文档的样式表列表，它会返回一个类似数组的对象，里面有 CSSStyleSheet 的对象，然后根据需要添加或移除样式。

  2. 内联样式

    • 直接在想要动态样式的元素上添加内联样式 —— HTMLElement.style 属性。

      如：para.style.color = "white";

  3. 类名操作（推荐）

    • 在单独的样式表中写样式，并通过添加或删除类名来引用这些样式（使用元素的 classList 的 add() 方法）。

📦 数组 ([,,,...]) - 索引集合

增删元素

  • push()：在数组末尾添加一个或多个项目。
  • unshift()：将项目添加到数组开头。
  • pop()：移除数组中最后一个项目。
  • shift()：从数组中移除第一个项。
  • **splice()**：根据索引移除项目。
    • 第一个参数：说明从哪里开始移除项目。
    • 第二个参数：说明应移除多少项目。

遍历与转换

  • **for...of...**：访问数组每一项。
  • **map()**：对数组中的每个项目做同样的操作，最终留下包含更改后物品的数组。
    • 给 map() 赋予一个函数，它会对数组中的每个项调用该函数一次。
  • **filter()**：创建一个新的数组，只包含原始数组中与某个测试匹配的项目。
    • 和 map() 一样，给 filter() 赋予一个函数，它对数组中的每个项目调用该函数。
  • **join()**：与 split() 相反，将数组连接为字符串。
  • **toString()**：数组转换为字符串的另一种方法。
    • join() 可以指定不同的分隔符。
    • toString() 总是用逗号分隔。

Hexo 图片加载问题排查与原理深度剖析

📝 问题背景

昨天我在我本地的 myblog/sources/_posts/ 写了一篇 blog (260315.md)。

  • 在 blog 中我加入了一张图片 naxida.jpg。
  • 并把该图片存到与 260315.md 同目录的 images 文件夹里。
  • 然后在根目录下执行 hexo clean、hexo g、hexo d 部署到 github pages。
  • 结果：我的 lnlikecutegirl.github.io 加载不出来图片。

🔍 尝试的解决方法

通过询问 AI，我得到如下解决方法及反馈：

✅ 方法 1：移动图片至源目录（成功）

  1. 将 naxida.jpg 从 source/_posts/images/ 移动到 source/images/
  2. 在 md 中使用 ![纳西妲](/images/naxida.jpg)
  3. 重新按 hexo clean && hexo g && hexo d

    结果：加载出来了。

❌ 方法 2：使用插件（失败）

  1. 安装一个名为 hexo-asset-image 的插件
  2. 在 md 中使用 ![纳西妲](./images/naxida.jpg) 或 ![纳西妲](_posts/images/naxida.jpg)

    结果：未加载出来。

⚙️ 原因剖析：Hexo 工作机制

1. hexo clean

核心作用：清除缓存文件和已生成的静态文件，确保下一次生成（hexo g）是从零开始，完全重新构建网站。

  • 删除 public/ 文件夹
    • 内容：这是 hexo g 生成的最终网站文件（HTML, CSS, JS, 图片等）。
  • 删除 db.json 文件
    • 内容：这是 Hexo 的数据库缓存文件。Hexo 为了加快生成速度，会把所有文章、标签、分类的信息存到这个 JSON 文件里。下次生成时，它只处理有变动的部分。
  • 删除 scaffolds/.gitignore (视情况)
    • 某些版本或配置下，它也会清理一些临时的 scaffold 缓存，但主要是前两项。

⚠️ 注意：它不会删除你的源码（source/_posts）、配置文件（_config.yml）、主题文件（themes）或 node_modules。

2. hexo g (Generate)

hexo g 时，Hexo 会对 _posts 里的 .md 文件做以下操作：

  1. 读取文件：读取 source/_posts/260315.md。
  2. 解析 Front-matter：读取文件头部的日期、标题等信息（例如 date: 2026-03-15）。
  3. 应用 permalink 规则：根据 _config.yml 中的 permalink 设置（默认通常是 :year/:month/:day/:title/），计算输出路径。
    • 输入：260315.md (日期 2026-03-15)
    • 输出路径计算：public/2026/03/15/260315/index.html
  4. 渲染 HTML：将 Markdown 内容转换成 HTML，填入主题模板，生成最终的 index.html 文件。

💡 核心规则：文件处理策略

Hexo 在生成静态网页时，只会“搬运”你放在 source 根目录下的文件，而会直接忽略掉 _posts 文件夹里除了 .md 文章文件以外的其他文件（如图片、视频等）。

Hexo 对 source 目录下的不同子文件夹有完全不同的“处理策略”。我们可以把 source 目录想象成一个工厂的原材料仓库，而 hexo g（生成）就是生产线。

  • 📂 _posts 文件夹

    • 规则：Hexo 认为 _posts 里只应该放文章草稿（.md 文件）。
    • 动作：输入 hexo g，它只从 _posts 里抓取 .md 文件，把它们“加工”成 HTML 网页。
    • 后果：对于 _posts 里的其他任何文件（图片、视频、txt 等），生产线会认为它们是“废料”或“无关文件”，直接忽略，不会搬运到最终的产品区（public 目录）。

  • 📂 source 根目录下的其他文件夹（如 images）

    • 规则：除了 _posts, _drafts, _data 等几个特殊文件夹外，source 根目录下的其他所有文件夹（比如你新建的 images, about, files 等），Hexo 都会执行“原样复制”策略。
    • 动作：看到 source/images/naxida.jpg，直接在最终产品区 public 里创建一个一模一样的文件夹结构：public/images/，把图片原封不动地复制过去：public/images/naxida.jpg。

3. hexo d (Deploy)

当你输入 hexo d 时，Hexo 首先会读取博客根目录下的 _config.yml 文件，寻找 deploy 部分的配置。

  • type: 告诉 Hexo 用什么方式部署（最常用的是 git）。
  • repo: 目标仓库的地址（通常是你的 GitHub Pages 仓库）。
  • branch: 要推送到哪个分支（GitHub Pages 通常是 main 或 master）。

执行步骤：

  1. 第一步：自动生成 (Generate)

    • 如果你之前没有运行过 hexo g，hexo d 通常会自动先执行生成操作。
    • 它会读取 source 文件夹里的文章和资源，渲染主题模板，生成完整的静态网站文件到 public 文件夹。
    • 此时，你的 public 文件夹里就是最终要发布的网站内容（HTML, CSS, JS, 图片等）。

  2. 第二步：准备 Git 仓库 (The Magic Step)

    • 这是最关键的一步，也是很多新手困惑的地方。
    • Hexo 会在你的博客目录下创建一个隐藏的文件夹，通常叫 .deploy_git（注意：它和你源码管理的 .git 是分开的！）。
    • 初始化：Hexo 会在 .deploy_git 里初始化一个全新的 Git 仓库。
    • 添加远程：它会把 _config.yml 里配置的 repo 地址添加为这个临时仓库的 origin。
    • 复制文件：它把刚刚生成的 public 文件夹里的所有内容，复制到这个 .deploy_git 文件夹中。
    • 提交更改：它会自动执行 git add . 和 git commit，提交信息通常默认为 "Site updated: [当前时间]"。

    • 💡 为什么要这么做？

    • 因为你的博客源码目录（包含 _posts, node_modules, _config.yml 等）非常杂乱，不能直接推到 GitHub Pages 仓库。GitHub Pages 只需要干净的静态文件（即 public 里的内容）。.deploy_git 就是一个“纯净的中间站”，只存放最终网站文件。

  3. 第三步：推送到远程 (Push)

    • 最后，Hexo 执行 git push 命令：将 .deploy_git 里的内容推送到你配置的 repo 地址的指定 branch 分支。

🗺️ 相对路径和绝对路径详解

目录结构对比

📂 my-blog (源码根目录 A)

【生成的网站 - 书店】(读者看见的)
📂 public (网站根目录 B)

路径定义

  • 相对路径 (Relative Path)
    • 定义：相对于当前文件所在位置的路径。
    • 特征：不以 / 开头，通常包含 . (当前目录) 或 .. (上一级目录)。

关键概念：当你在 Markdown 里写 ![图](/images/pic.jpg) 时，这个 / 指的是 **网站根目录 (Public Root)**，而不是你电脑上的源码根目录！
即：/images/pic.jpg = public/images/pic.jpg

a. ![纳西妲](./images/naxida.jpg)
