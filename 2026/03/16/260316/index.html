Hexo 图片加载问题排查与原理深度剖析

📝 问题背景

昨天我在我本地的 myblog/sources/_posts/ 写了一篇 blog (260315.md)。

  • 在 blog 中我加入了一张图片 naxida.jpg。
  • 并把该图片存到与 260315.md 同目录的 images 文件夹里。
  • 然后在根目录下执行 hexo clean、hexo g、hexo d 部署到 github pages。
  • 结果：我的 lnlikecutegirl.github.io 加载不出来图片。

🔍 尝试的解决方法

通过询问 AI，我得到如下解决方法及反馈：

✅ 方法 1：移动图片至源目录（成功）

  1. 将 naxida.jpg 从 source/_posts/images/ 移动到 source/images/
  2. 在 md 中使用 ![纳西妲](/images/naxida.jpg)
  3. 重新按 hexo clean && hexo g && hexo d

    结果：加载出来了。

❌ 方法 2：使用插件（失败）

  1. 安装一个名为 hexo-asset-image 的插件
  2. 在 md 中使用 ![纳西妲](./images/naxida.jpg) 或 ![纳西妲](_posts/images/naxida.jpg)

    结果：未加载出来。

⚙️ 原因剖析：Hexo 工作机制

1. hexo clean

核心作用：清除缓存文件和已生成的静态文件，确保下一次生成（hexo g）是从零开始，完全重新构建网站。

  • 删除 public/ 文件夹
    • 内容：这是 hexo g 生成的最终网站文件（HTML, CSS, JS, 图片等）。
  • 删除 db.json 文件
    • 内容：这是 Hexo 的数据库缓存文件。Hexo 为了加快生成速度，会把所有文章、标签、分类的信息存到这个 JSON 文件里。下次生成时，它只处理有变动的部分。
  • 删除 scaffolds/.gitignore (视情况)
    • 某些版本或配置下，它也会清理一些临时的 scaffold 缓存，但主要是前两项。

⚠️ 注意：它不会删除你的源码（source/_posts）、配置文件（_config.yml）、主题文件（themes）或 node_modules。

2. hexo g (Generate)

hexo g 时，Hexo 会对 _posts 里的 .md 文件做以下操作：

  1. 读取文件：读取 source/_posts/260315.md。
  2. 解析 Front-matter：读取文件头部的日期、标题等信息（例如 date: 2026-03-15）。
  3. 应用 permalink 规则：根据 _config.yml 中的 permalink 设置（默认通常是 :year/:month/:day/:title/），计算输出路径。
    • 输入：260315.md (日期 2026-03-15)
    • 输出路径计算：public/2026/03/15/260315/index.html
  4. 渲染 HTML：将 Markdown 内容转换成 HTML，填入主题模板，生成最终的 index.html 文件。

💡 核心规则：文件处理策略

Hexo 在生成静态网页时，只会“搬运”你放在 source 根目录下的文件，而会直接忽略掉 _posts 文件夹里除了 .md 文章文件以外的其他文件（如图片、视频等）。

Hexo 对 source 目录下的不同子文件夹有完全不同的“处理策略”。我们可以把 source 目录想象成一个工厂的原材料仓库，而 hexo g（生成）就是生产线。

  • 📂 _posts 文件夹

    • 规则：Hexo 认为 _posts 里只应该放文章草稿（.md 文件）。
    • 动作：输入 hexo g，它只从 _posts 里抓取 .md 文件，把它们“加工”成 HTML 网页。
    • 后果：对于 _posts 里的其他任何文件（图片、视频、txt 等），生产线会认为它们是“废料”或“无关文件”，直接忽略，不会搬运到最终的产品区（public 目录）。

  • 📂 source 根目录下的其他文件夹（如 images）

    • 规则：除了 _posts, _drafts, _data 等几个特殊文件夹外，source 根目录下的其他所有文件夹（比如你新建的 images, about, files 等），Hexo 都会执行“原样复制”策略。
    • 动作：看到 source/images/naxida.jpg，直接在最终产品区 public 里创建一个一模一样的文件夹结构：public/images/，把图片原封不动地复制过去：public/images/naxida.jpg。

3. hexo d (Deploy)

当你输入 hexo d 时，Hexo 首先会读取博客根目录下的 _config.yml 文件，寻找 deploy 部分的配置。

  • type: 告诉 Hexo 用什么方式部署（最常用的是 git）。
  • repo: 目标仓库的地址（通常是你的 GitHub Pages 仓库）。
  • branch: 要推送到哪个分支（GitHub Pages 通常是 main 或 master）。

执行步骤：

  1. 第一步：自动生成 (Generate)

    • 如果你之前没有运行过 hexo g，hexo d 通常会自动先执行生成操作。
    • 它会读取 source 文件夹里的文章和资源，渲染主题模板，生成完整的静态网站文件到 public 文件夹。
    • 此时，你的 public 文件夹里就是最终要发布的网站内容（HTML, CSS, JS, 图片等）。

  2. 第二步：准备 Git 仓库 (The Magic Step)

    • 这是最关键的一步，也是很多新手困惑的地方。
    • Hexo 会在你的博客目录下创建一个隐藏的文件夹，通常叫 .deploy_git（注意：它和你源码管理的 .git 是分开的！）。
    • 初始化：Hexo 会在 .deploy_git 里初始化一个全新的 Git 仓库。
    • 添加远程：它会把 _config.yml 里配置的 repo 地址添加为这个临时仓库的 origin。
    • 复制文件：它把刚刚生成的 public 文件夹里的所有内容，复制到这个 .deploy_git 文件夹中。
    • 提交更改：它会自动执行 git add . 和 git commit，提交信息通常默认为 "Site updated: [当前时间]"。

    • 💡 为什么要这么做？

    • 因为你的博客源码目录（包含 _posts, node_modules, _config.yml 等）非常杂乱，不能直接推到 GitHub Pages 仓库。GitHub Pages 只需要干净的静态文件（即 public 里的内容）。.deploy_git 就是一个“纯净的中间站”，只存放最终网站文件。

  3. 第三步：推送到远程 (Push)

    • 最后，Hexo 执行 git push 命令：将 .deploy_git 里的内容推送到你配置的 repo 地址的指定 branch 分支。

🗺️ 相对路径和绝对路径详解

目录结构对比

📂 my-blog (源码根目录 A)

【生成的网站 - 书店】(读者看见的)
📂 public (网站根目录 B)

路径定义

  • 相对路径 (Relative Path)
    • 定义：相对于当前文件所在位置的路径。
    • 特征：不以 / 开头，通常包含 . (当前目录) 或 .. (上一级目录)。

关键概念：当你在 Markdown 里写 ![图](/images/pic.jpg) 时，这个 / 指的是 **网站根目录 (Public Root)**，而不是你电脑上的源码根目录！
即：/images/pic.jpg = public/images/pic.jpg

错误写法分析

a. ![纳西妲](./images/naxida.jpg)

  • 你的意图：你想说：“图片就在当前文章旁边的 images 文件夹里。”通常这意味着你在 source/_posts/ 旁边建了一个 images 文件夹，或者在文章同级的文件夹里。
  • Hexo 的默认行为（坑点所在！）：
    • 默认情况下，Hexo 不会把你文章旁边的 images 文件夹复制到 public 目录中对应的位置。
  • 路径推演：
    • 源码结构：
    • **生成后的结构 (public)**：
    • 结果：浏览器去 public/2026/03/16/my-article/images/naxida.jpg 找图片，找不到，404 错误。

b. ![纳西妲](_posts/images/naxida.jpg)

  • 你的意图：你可能在想：“我在 source/_posts 里写文章，图片在 source/_posts/images 里，所以我应该写 _posts/images/... 来指向它。”
  • 为什么这是错的？
    • 这是一个典型的“源码思维”陷阱。
    • **浏览器看不见 source**：网站发布后，source 文件夹彻底消失了。浏览器根本不知道 _posts 是个什么东西。
